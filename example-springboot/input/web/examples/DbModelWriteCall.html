Write Example

With ==>{ConfigLinkCall->ModelConfig, DbModelWriteCall}. one has the possibility to write entries in the database on the server side depending on the object parameters defined in ==>{ConfigLinkCall->FieldConfig, sourcePath}..

In the example the ==>{ConfigLinkCall->FieldConfig, sourcePath}. "abc" contains an ==>{ConfigLinkCall->ModelConfig, AnObject}. object with ==>{ConfigLinkCall->FieldConfig, id}. "3". ==>{ConfigLinkCall->ModelConfig, DbModelWriteCall}. will update the third entry of the preloaded ==>{GithubLinkCall->NONE, EODBR||DbSqlConfig.json#L43}.. The updated entry will be returned on ==>{ConfigLinkCall->FieldConfig, targetPath}. "xyz".

The examples would not work since the role guest has not the permission to insert or update. Only ==>{ConfigLinkCall->ModelConfig, DbModelReadCall}. is allowed.

The configuration of ==>{ConfigLinkCall->ModelConfig, AnObject}. is a special extension of ==>{ConfigLinkCall->ModelConfig, ModelConfigObject}.. Any object with a configuration of type ==>{ConfigLinkCall->ModelConfig, ModelConfigDbObject}. could be used for a generic ==>{ConfigLinkCall->ModelConfig, DbModelCall}.

This allows to define properties like ==>{ConfigLinkCall->FieldConfig, table}. with ==>{ConfigLinkCall->ModelConfig, ModelConfigDbProperties}.

With the implementation of ==>{ConfigLinkCall->ModelConfig, PermissionProperties}. the model configuration has some role access restrictions.

==>{ConfigLinkCall->ModelConfig, AnObject}. shows that there is a only read property defined for "guest".
